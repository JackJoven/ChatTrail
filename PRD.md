# PRD: ChatTrail 全平台 AI 会话增强插件

版本：v0.1
日期：2026-04-18
状态：草案
产品代号：ChatTrail

## 1. 背景与问题

Gemini Voyager 已经验证了一类明确需求：AI 用户需要一个跨会话、跨设备、可管理、可导出、可复用的“AI 会话工作台”。它围绕 Google Gemini 和 AI Studio 提供了时间轴导航、文件夹管理、提示词库、云同步、公式复制、聊天导出、Mermaid 渲染、批量删除、引用回复、默认模型、标签页标题同步等增强能力。

当前痛点是这些能力被绑定在单一或少数平台上。真实用户往往同时使用 ChatGPT、Claude、豆包、通义千问、Grok、智谱清言、Gemini、AI Studio、DeepSeek、Poe、Kimi 等多个 AI 产品。不同平台各自有会话、历史记录、侧边栏、模型选择、导出、提示词和文件能力，但体验不统一，数据难迁移，提示词资产分散，长对话难定位，关键上下文难沉淀。

ChatTrail 的目标是把 Voyager 式增强能力抽象成一套“平台适配器 + 统一能力层”的浏览器扩展，让用户在主流 AI Web 产品中获得尽量一致的管理、导航、导出和复用体验。

## 2. 产品目标

1. 在多 AI 平台上提供统一的会话组织、提示词复用、长对话导航、引用回复、内容导出和渲染增强能力。
2. 建立平台适配器架构，新增平台时只需要实现该平台的 DOM 解析、会话识别、输入框控制、消息抽取、侧边栏操作等适配接口。
3. 将“能跨平台稳定实现”的能力优先产品化，将“高度依赖平台私有 UI 的能力”做成分级支持和降级体验。
4. 用户数据默认本地优先，支持可选云同步和本地备份，避免把会话内容发送到第三方服务器。
5. 形成可持续维护的兼容性矩阵和自动化回归机制，降低 AI 平台频繁改版带来的维护成本。

## 3. 非目标

1. 不绕过各 AI 平台的登录、订阅、模型权限、风控和访问限制。
2. 不替用户批量抓取或迁移违反平台条款的数据。
3. 不承诺所有平台在首版实现 100% 功能等价；PRD 中定义功能分级和降级策略。
4. 不在首版实现自建 AI 聊天后端；ChatTrail 是增强现有 Web 产品体验的浏览器扩展。
5. 不在首版支持移动端 App 内增强；移动端只考虑浏览器 Web 访问的可行性验证。

## 4. 目标用户

核心用户：

- 重度 AI 用户：每天在多个 AI 平台间切换，需要整理大量会话和提示词。
- 研究、写作、学习用户：长对话多，需要时间轴、书签、导出、引用回复和知识归档。
- 开发者：经常保存代码片段、提示词模板、Mermaid 图、公式、调试记录，需要结构化导出。
- 团队/个人知识管理用户：希望把 AI 讨论沉淀到 Obsidian、Notion、Markdown、PDF 或 JSON。

次级用户：

- 只使用一个 AI 平台，但不满意原生会话管理能力的用户。
- 希望把提示词库带到任意网站输入框的用户。

## 5. 平台范围

首批目标平台分三期推进：

P0 平台：

- ChatGPT：`chatgpt.com`
- Claude：`claude.ai`
- Gemini：`gemini.google.com`
- Google AI Studio：`aistudio.google.com`

P1 平台：

- 豆包：以实际官方 Web 域名为准，启动适配前确认。
- 通义千问 / Qwen：候选域名包括 `tongyi.aliyun.com/qianwen/`、`qianwen.aliyun.com`、`chat.qwen.ai`，启动适配前确认具体入口。
- Grok：`grok.com`，可扩展到 X 内嵌 Grok。
- 智谱清言 / ChatGLM：`chatglm.cn`

P2 平台：

- DeepSeek、Kimi、Poe、Microsoft Copilot、Perplexity、文心一言、腾讯元宝、秘塔 AI 搜索等。

平台进入标准：

- Web 端有稳定会话页面。
- 能通过浏览器扩展注入脚本。
- 能可靠识别会话 ID、消息列表、用户/助手消息、输入框、侧边栏或历史列表中的至少一部分。
- 不需要绕过登录、付费、验证码或平台安全限制。

## 6. 核心产品原则

1. 能本地完成就本地完成：解析、渲染、导出、提示词插入、公式复制默认在浏览器侧完成。
2. 能统一就统一：提示词库、导出格式、文件夹数据模型、设置面板、快捷键体系尽量跨平台一致。
3. 能降级就降级：如果某平台无法批量删除，则显示“不支持批量删除”，但仍保留导出、提示词、引用回复等能力。
4. 不破坏原站：所有增强 UI 必须可关闭，不遮挡核心输入和阅读流程。
5. 透明兼容性：用户在设置页能看到每个平台当前支持哪些能力、哪些能力因平台限制不可用。

## 7. 功能分级

### 7.1 通用能力

这些能力应尽量覆盖所有目标平台：

- 提示词库：保存、搜索、标签、分类、插入、导入、导出。
- 引用回复：选中对话内容，一键以 Markdown 引用格式插入输入框。
- 聊天导出：导出当前会话为 Markdown、JSON、PDF。
- 公式复制：识别 LaTeX 或 MathJax/KaTeX 节点，复制 LaTeX 和 MathML。
- Mermaid 渲染：识别代码块并渲染图表，支持源码/图表切换和全屏查看。
- 标签页标题同步：将浏览器标签页标题同步为当前会话标题。
- 消息时间戳：记录用户发送时间，展示在消息旁或 hover 面板中。
- 视觉效果：雪、雨、樱花等轻量 overlay，默认关闭。
- 字体大小和聊天宽度：在平台允许的范围内调整阅读体验。
- 本地备份与恢复：导出/导入全部设置、提示词、文件夹、星标、时间戳等。

### 7.2 高价值平台适配能力

这些能力依赖平台 DOM 和侧边栏结构，需要逐平台实现：

- 时间轴导航：消息节点、hover 预览、点击跳转、键盘导航、星标节点、层级折叠。
- 文件夹管理：将原平台历史会话组织到自定义文件夹中，支持拖拽、重命名、多选、颜色、嵌套。
- 批量移动：多选会话后移动到文件夹。
- 批量删除：仅在平台 UI 可安全触发删除且能提供确认时支持。
- 默认模型：仅在平台有可识别模型选择器时支持。
- 隐藏最近记录/原生侧边栏项：仅在不会破坏平台功能时支持。
- 防自动滚动：在长对话阅读中阻止发送后强制跳到底部。
- 输入框折叠：输入区为空且失焦时折叠，减少阅读遮挡。
- 侧边栏宽度调整：对有侧边栏的平台支持宽度调节。
- 会话分叉：从指定消息节点复制上文并创建新会话，平台支持新会话和自动填入时启用。

### 7.3 平台专属能力

这些能力不强行全平台等价，而是按平台能力实现：

- Deep Research / 深度研究导出：支持有可识别研究过程的平台，导出思考阶段、引用来源、搜索链接和阶段摘要。
- 图片水印处理：仅针对平台生成图片且合法可处理的场景，默认不做跨平台泛化。
- IDE 上下文同步：开发者功能，先支持 ChatGPT、Claude、Gemini，再扩展到其他平台。
- 原生项目/Artifact/Canvas/智能体管理：按平台插件化扩展，不纳入首版主干。

## 8. MVP 范围

MVP 覆盖 P0 平台中的 ChatGPT、Claude、Gemini，并预留 AI Studio。目标是在 8 到 10 周内完成一个可安装、可日常使用的浏览器扩展。

MVP 必须包含：

1. 平台适配器基础框架。
2. 统一设置面板和平台能力矩阵。
3. 提示词库：保存、标签、搜索、插入、导入、导出。
4. 时间轴导航：ChatGPT、Claude、Gemini。
5. 当前会话导出：Markdown、JSON；PDF 可作为增强项。
6. 引用回复：ChatGPT、Claude、Gemini。
7. Mermaid 渲染：ChatGPT、Claude、Gemini。
8. 公式复制：ChatGPT、Claude、Gemini。
9. 标签页标题同步。
10. 本地备份与恢复。

MVP 可延期：

- 文件夹拖拽管理。
- 批量删除。
- 默认模型。
- 云同步。
- 会话分叉。
- 豆包、通义千问、Grok、智谱清言完整适配。

## 9. 功能需求

### 9.1 平台适配器

每个平台适配器必须实现以下接口中的可行子集：

- `detect()`：判断当前页面是否属于该平台。
- `getConversationId()`：获取当前会话唯一 ID。
- `getConversationTitle()`：获取当前会话标题。
- `getMessages()`：返回标准消息列表。
- `getMessageAnchor(messageId)`：返回消息节点，用于跳转和高亮。
- `getInputController()`：提供插入文本、聚焦、读取输入内容能力。
- `getSidebarController()`：提供历史列表读取、会话项定位、拖拽/移动/删除等能力。
- `getModelController()`：提供模型列表读取和默认模型设置能力。
- `observeChanges()`：监听消息、标题、侧边栏变化。

标准消息模型：

```ts
type ChatMessage = {
  id: string;
  role: "user" | "assistant" | "system" | "tool" | "unknown";
  text: string;
  html?: string;
  createdAt?: string;
  attachments?: Attachment[];
  codeBlocks?: CodeBlock[];
  citations?: Citation[];
};
```

### 9.2 提示词库

用户故事：

- 作为用户，我可以在任意支持平台保存当前输入框内容为提示词。
- 作为用户，我可以给提示词添加标签、标题、说明和适用场景。
- 作为用户，我可以通过关键词或标签快速搜索提示词并一键插入。
- 作为用户，我可以导入/导出提示词，避免被插件或浏览器锁定。

验收标准：

- 保存提示词不依赖远程服务。
- 搜索 1000 条提示词时，结果响应时间小于 200ms。
- 插入提示词后输入框聚焦，原有输入内容不被误删，除非用户选择替换。
- 支持 JSON 导入/导出，后续兼容 Voyager prompt 格式。

### 9.3 文件夹管理

用户故事：

- 作为用户，我可以把不同平台的会话放入自定义文件夹。
- 作为用户，我可以在同一平台内使用拖拽、多选、重命名和颜色来组织会话。
- 作为用户，我可以选择按平台隔离文件夹，也可以开启跨平台聚合视图。

验收标准：

- 文件夹至少支持二级层级；内部数据模型支持更多层级。
- 拖拽失败时不改变本地数据。
- 同一会话在同一平台只允许出现于一个主文件夹，可支持标签式多归属作为后续功能。
- 平台无法修改原生侧边栏时，使用 ChatTrail 自建侧边栏面板展示。

### 9.4 时间轴导航

用户故事：

- 作为用户，我可以在长对话右侧看到消息节点，并点击跳转。
- 作为用户，我可以 hover 节点预览消息摘要。
- 作为用户，我可以给关键消息加星标。
- 作为用户，我可以用快捷键跳到上一条/下一条消息。

验收标准：

- 100 条消息以内构建时间轴耗时小于 500ms。
- 节点点击后滚动到对应消息，并提供短暂高亮。
- 页面新增消息后时间轴自动增量更新。
- 快捷键可自定义，且在输入框聚焦时不误触发导航。

### 9.5 聊天导出

用户故事：

- 作为用户，我可以把当前会话导出为 Markdown，用于 Obsidian、Notion 或归档。
- 作为用户，我可以导出 JSON，供开发者二次处理。
- 作为用户，我可以导出 PDF，用于分享和打印。

验收标准：

- Markdown 保留用户/助手角色、代码块、表格、链接和基本引用。
- JSON 使用标准消息模型，包含平台、会话 ID、标题、导出时间。
- PDF 至少支持浏览器打印模式，图片提取按平台能力尽力支持。
- 导出过程不上传会话内容到第三方服务。

### 9.6 引用回复

用户故事：

- 作为用户，我可以选中任意消息片段，一键插入引用到输入框。

验收标准：

- 选中文本后显示浮动按钮。
- 点击后以 Markdown blockquote 格式插入。
- 多行文本每行自动加 `>`。
- 不在输入框、设置面板、插件弹窗中误触发。

### 9.7 Mermaid 渲染

用户故事：

- 作为用户，我希望 AI 输出 Mermaid 代码块后自动变成图，而不是手动复制到别处渲染。

验收标准：

- 支持 flowchart、sequenceDiagram、gantt、classDiagram、stateDiagram、pie、erDiagram 等主流类型。
- 支持源码/图表切换。
- 支持全屏查看、缩放、拖拽平移。
- 渲染失败时保留原始代码块，并展示错误提示。

### 9.8 公式复制

用户故事：

- 作为用户，我可以一键复制公式的 LaTeX 或 Word 可编辑 MathML。

验收标准：

- 识别页面中的 KaTeX、MathJax、原始 LaTeX 文本。
- hover 公式时显示复制按钮。
- 支持复制 LaTeX 和 MathML。
- 复制失败时给出明确反馈。

### 9.9 云同步

首版本地优先，云同步作为 P1。

用户故事：

- 作为用户，我可以把提示词、文件夹、设置同步到自己的云端空间。

候选方案：

- 浏览器 `chrome.storage.sync`：容量有限，适合设置和少量提示词。
- 用户自有云盘：Google Drive、OneDrive、坚果云 WebDAV。
- 自建同步服务：长期可行，但会显著增加隐私、安全和运维成本。

验收标准：

- 云同步默认关闭。
- 同步前展示数据类型和目标位置。
- 支持手动上传、下载合并、冲突提示。
- 不同步完整聊天内容，除非用户明确选择。

## 12. 权限与隐私

浏览器权限原则：

- 按需请求站点权限，不一次性请求所有网站。
- 用户启用某平台时再请求对应域名权限。
- 提示词库在自定义网站启用时，要求用户显式添加网站。

数据存储：

- 本地数据使用 IndexedDB 存储提示词、文件夹、星标、时间戳、导出元数据。
- 设置使用 browser storage。
- 敏感会话内容默认不长期保存，除非用户开启导出、收藏、摘要或本地归档。

隐私要求：

- 不采集完整会话内容用于分析。
- 不向自有服务器发送用户输入、AI 回复、文件内容。
- 崩溃日志和匿名统计必须可关闭，且默认不包含会话正文。
- 云同步必须展示同步内容范围。

## 13. 技术方案概要

推荐技术栈：

- Extension Manifest V3。
- TypeScript。
- Vite + WebExtension 构建。
- React 或 Svelte 用于 Popup、Options、增强面板。
- IndexedDB + Dexie 用于本地数据库。
- Web Components 或 Shadow DOM 隔离注入 UI。
- Playwright 做平台 UI 回归测试。

核心模块：

- `platform-adapters`：各平台适配器。
- `capability-core`：时间轴、导出、提示词、渲染、公式等通用能力。
- `storage`：本地存储、备份、迁移、同步。
- `ui-injection`：页面注入、浮动按钮、侧边栏、overlay。
- `settings`：功能开关、平台权限、快捷键。
- `compatibility-tests`：平台 DOM 探测和截图回归。

适配器策略：

- 不直接把功能写死到平台脚本中。
- 每个能力只依赖标准接口，例如时间轴只依赖 `getMessages()` 和 `getMessageAnchor()`。
- 每个平台适配器维护 selector 版本号和探测用例。
- 当 selector 失效时，功能自动降级并提示用户提交诊断信息。

## 14. 数据模型

核心实体：

- `Platform`：平台 ID、名称、域名、支持能力、适配器版本。
- `ConversationRef`：平台、会话 ID、标题、URL、最后访问时间。
- `Folder`：ID、平台范围、名称、父级、颜色、排序。
- `FolderAssignment`：会话与文件夹关系。
- `Prompt`：标题、正文、标签、分类、变量、创建/更新时间。
- `TimelineMark`：会话 ID、消息 ID、星标、等级、备注。
- `MessageTimestamp`：会话 ID、消息 ID、发送时间。
- `Settings`：全局设置、平台设置、快捷键、功能开关。

## 17. 风险与应对

平台 DOM 频繁变化：

- 应对：适配器分层、自动探测、截图回归、版本化 selector、降级提示。

各平台功能差异过大：

- 应对：功能矩阵公开展示，使用“统一核心 + 平台增强”的产品策略。

权限请求导致用户不信任：

- 应对：按平台逐个授权，解释权限用途，默认不读取未启用站点。

导出内容涉及隐私：

- 应对：全部本地处理，导出前提示范围，不做默认云端保存。

批量删除存在误删风险：

- 应对：P2 延后；必须有二次确认、数量限制、进度反馈和平台原生删除成功确认。

使用 Voyager 代码的许可证风险：

- 应对：Gemini Voyager 使用 GPL-3.0。若 fork 或复用其代码，ChatTrail 也需要按 GPL-3.0 兼容方式开源并保留版权与第三方声明。若希望闭源或商业闭源，应仅参考产品思路，重新实现代码。

## 18. 验收清单

MVP 发布前必须满足：

- Chrome/Edge 可安装开发包。
- ChatGPT、Claude、Gemini 三个平台能正确识别当前会话。
- 三个平台都能插入提示词。
- 三个平台都能展示时间轴并跳转消息。
- 三个平台都能导出 Markdown 和 JSON。
- 三个平台都能使用引用回复。
- Mermaid 和公式复制至少在标准代码块/公式输出中可用。
- 设置页能展示平台支持状态。
- 本地备份和恢复可用。
- 不启用某平台时，不注入该平台脚本。

## 19. 开放问题

1. 产品最终是否基于 Gemini Voyager fork？如果 fork，需要接受 GPL-3.0 约束。
2. 是否计划上架 Chrome Web Store、Edge Add-ons、Firefox Add-ons、Safari？不同商店审核和权限策略不同。
3. 是否需要团队协作版，例如团队提示词库、共享文件夹、统一配置？
4. 云同步首选用户自有云盘还是自建账号系统？
5. 是否支持国内浏览器扩展市场，以及是否需要 Manifest V2/V3 双构建？
6. 是否要把 DeepSeek、Kimi 纳入 P1，而不是 P2？
